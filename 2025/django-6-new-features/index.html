Upcoming Django 6.0 Features

November 12, 2025

The 6.0 release of Django is scheduled to release next month and I wanted to write about the highlighted features, which I’m looking forward to using.

1. Background Tasks

The upcoming Tasks API defines an interface for interacting with objects that are expected to run outside of the typical HTTP request/response cycle, like sending emails. For example, there is a TaskResult class for obtaining task progress and return values.

Interestingly, the Django team is drawing an explicit line at providing task runtimes; Django 6.0 defines a common API for queuing tasks and obtaining results but does not prescribe how these should be run, instead allowing providers to “plug in” via the familiar BACKEND setting.

Reading the Django Enhancement proposal (DEP 14), the main motivation for this is clear: "[…] the ecosystem is filled with incredibly popular frameworks, all of which interact with Django in slightly different ways […]"

It’s difficult to discuss this feature without celery coming to mind… It’s the de-facto standard for defining and actually running tasks for Django projects, as far as I’m aware. I’ve used Celery extensively (it’s part of my go-to tooling for Django projects) and I’m curious to see how this standardized interface will be adopted by them, if at all. I’d expect either the celery project itself or someone else to write a celery-compatible Task backend (or wrapper) in the near future…

2. Template Partials

Built-in support for template partials is also coming in Django 6.0. I think this improvement to Django’s templates component is particularly welcome in light of the increase in htmx adoption for Django projects, as this year’s Django Developer Survey results show.

Not to be confused with template inheritance or template inlining via includes, template “partials” are intended for small HTML re-use via the new {% partialdef %} and {% partial %} tags.

The ability to directly access named partials is likely to become a best-practice when it comes to working with htmx-style “respond with an HTML snippet” requests.

This feature is essentially “vendoring-in” the django-template-partials third-party package.

3. Content Security Policy

I’ll quickly mention the other highlighted feature from Django’s 6.0 release notes: built-in Content Security Policy support. This is essentially “vendoring-in” the django-csp package (that’s two now), and is another notch in Django’s “batteries-included” belt.

Defining and maintaining a CSP is an important part of production readiness, so it’s great to see that Django is directly supporting this. Setting up a proper CSP protects against injections (vulnerability #5 in this year’s OWASP top 10).

How to Use These Features Today

You can opt-in to start testing these features today. The latest pre-release for the 6.0 branch is Django 6.0 beta 1, tagged as 6.0b1.

Install it using uv:

uv add "django==6.0b1"

or using pip:

pip install --pre django

Make sure to read through the deprecations and feature removals beforehand!

There are a bunch of other features and improvements coming besides the ones highlighted above, make sure to check out the release docs. Django 6.0 is set to be publicly released on December 3rd, 2025.
